# Validation Framework

Post-training validation checks for ML-IAM XGBoost predictions.

## Terminology

- **AR6 test set** (`y_test`) — the withheld 10% of AR6 scenarios used to evaluate
  model accuracy. This is real, observed IAM data. It is used in some checks as a
  reference to derive empirical bounds or thresholds, but it is **never** the thing
  being validated.
- **Predictions** (`preds`) — the synthetic scenarios generated by the model. These
  are what every check is evaluating. Even though the model inputs came from AR6
  scenarios, the predicted values are the model's own output and are treated as novel
  generated scenarios throughout this framework.

All checks load both `preds` and `y_test` from a completed run, but the question
being asked is always: **are the generated predictions physically plausible?**

## Usage

Run all checks at once:
```bash
python val_framework/run_all.py --run_id xgb_03
```

Run an individual check:
```bash
python val_framework/check_plausibility.py --run_id xgb_03
python val_framework/sum_check.py --run_id xgb_03
```

## Checks

### `check_plausibility.py` — Growth Rate Plausibility Check

**What it does:**
For each **generated (predicted) trajectory** (scenario × region × variable),
computes the 5-year period-on-period growth rate:

```
g_y = (x_y - x_{y-5}) / x_{y-5}
```

The AR6 test-set ground truth is used solely to **derive empirical reference bounds**
(by default the 1st/99th percentiles of observed growth rates per variable). These
bounds represent what physically plausible growth looks like in real IAM data. The
check then flags any timestep in the **generated predictions** where the growth rate
falls outside those bounds.

**Required model outputs:** Any combination of output variables works — the check
runs on whichever variables the model was trained to predict. However, the check
is only meaningful for variables that change over time. Constant or near-constant
variables will trivially pass with a growth rate of zero and contribute nothing
useful to the analysis.

**Required data:** The AR6 test-set ground truth is loaded automatically from the
cached `processed_data.parquet` for the run — used only to derive the reference
bounds, not as the thing being checked.

**Required data structure:**
- Each scenario × region × variable trajectory must have **at least 2 timesteps**
  in the test set for any growth rate to be computed. Trajectories with only 1
  timestep are silently skipped.
- Timesteps where the lagged value (`x_{y-5}`) is zero or missing are dropped
  (division by zero). Variables that are zero for extended periods — e.g. Solar or
  Wind in early years of low-ambition scenarios — will have those timesteps excluded
  from both the bounds derivation and the violation count. The printed output reports
  exactly how many rows were dropped for this reason.

**Key options:**
| Flag | Default | Description |
|---|---|---|
| `--percentile` | `1.0` | Lower tail percentile (upper = 100 - percentile). Use `5` for stricter bounds. |
| `--by_category` | off | Also break down violations by scenario category (C1–C8) |

**Outputs** (`results/xgb/<run_id>/plausibility/`):
- `report.txt` — full printed report
- `growth_rate_violations.csv` — every timestep with growth rate, bounds, violation flag, and severity
- `empirical_bounds.csv` — the derived lower/upper bounds per variable

### `sum_check.py` — Secondary Energy Electricity Sum Check

**What it does:**
Verifies that the **generated predictions** for `Secondary Energy|Electricity`
(total electricity generation) equal the sum of the predicted sub-fuel components
at every timestep:

```
sum = Biomass + Coal + Gas + Geothermal + Hydro + Nuclear + Oil + Solar + Wind
error = |total - sum| / |total|
```

A scenario **passes** if `error < 1.2%` at every timestep. This mirrors the
standard vetting metric used in the AR6 Scenarios Database.

**Required model outputs** (all of the following must be in the run's target variables):
- `Secondary Energy|Electricity` — total electricity generation
- `Secondary Energy|Electricity|Biomass`
- `Secondary Energy|Electricity|Coal`
- `Secondary Energy|Electricity|Gas`
- `Secondary Energy|Electricity|Geothermal`
- `Secondary Energy|Electricity|Hydro`
- `Secondary Energy|Electricity|Nuclear`
- `Secondary Energy|Electricity|Oil`
- `Secondary Energy|Electricity|Solar`
- `Secondary Energy|Electricity|Wind`

The script will exit with an informative error if any are missing from the run's targets.

**Required data:** Only the run's cached predictions — no AR6 ground truth is used
in this check. The sum constraint is a definitional identity that the generated
values must satisfy on their own.

**Key options:**
| Flag | Default | Description |
|---|---|---|
| `--threshold` | `0.012` | Maximum allowed relative error per timestep (1.2%) |

**Outputs** (`results/xgb/<run_id>/sum_check/`):
- `report.txt` — full printed report
- `scenario_summary.csv` — one row per scenario with mean error, max error, pass/fail
- `timestep_errors.csv` — one row per scenario × year with raw error values

## Adding a New Check

1. Create `val_framework/<your_check>.py` with a `main()` function that uses
   `argparse` and accepts at minimum `--run_id`.
2. Add an entry to the `CHECKS` registry in `run_all.py`:
   ```python
   CHECKS = [
       ("check_plausibility", "Growth rate plausibility check"),
       ("sum_check",          "Secondary Energy|Electricity sum check"),
       ("your_check",         "Description of your check"),   # <-- add here
   ]
   ```
3. Add a corresponding `--no-<your_check>` flag to `run_all.py` if you want
   it to be skippable.
4. Document your check in this README following the same structure as the
   existing checks. Be explicit about whether your check uses the AR6 test-set
   ground truth (as a reference only) or purely the generated predictions.
   The thing being validated is always `preds`. Specifically, make sure to include:
   - **What it does** — the metric or criterion being evaluated
   - **Required model outputs** — which variables must be present in the run's
     targets for the check to work (be explicit — a missing variable will cause
     a silent wrong result or a confusing crash for the next person)
   - **Required data** — any external files or resources beyond the run's
     cached predictions
   - **Key options** — flags and their defaults
   - **Outputs** — what files are saved and where
